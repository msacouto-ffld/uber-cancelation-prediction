## Problem Statement

Ride-hailing platforms face significant operational inefficiencies when bookings are cancelled. Cancellations create customer dissatisfaction, driver idle time, and reduced platform efficiency. This project builds a machine learning model that **predicts ride cancellations at booking time**, using only features available before the ride begins, so the platform can intervene proactively.

**Target variable:** `Cancelled = 1` if Booking Status ≠ Completed, else 0  
**Primary metrics:** ROC-AUC, Recall (Cancelled class)

---

## Results

| Model | ROC-AUC | Recall (Cancelled) | F1 (Cancelled) |
|---|---|---|---|
| Logistic Regression | 0.707 | 0.527 | 0.539 |
| Random Forest | 0.710 | 0.305 | 0.437 |
| Gradient Boosting (default) | 0.722 | 0.656 | 0.576 |
| **Gradient Boosting (tuned)** | **0.722** | **0.667** | **0.580** |

Best model: `HistGradientBoostingClassifier` tuned with `RandomizedSearchCV` over a time-based cross-validation split. Final hyperparameters are:
- 'classifier__min_samples_leaf': 20,
- 'classifier__max_iter': 100,
- 'classifier__max_depth': 4,
- 'classifier__learning_rate': 0.03,
- 'classifier__l2_regularization': 0.1.

The model's predictive power is concentrated in driver-assignment signals (`Avg VTAT`, `vtat_missing`). A second feature set adds behavioral history per customer and location (`customer_cancel_rate`, `pickup_risk`, `drop_risk`, `route_risk`) computed without leakage using expanding-window past data only. Features known only after ride completion (booking value, ride distance, ratings, payment method) were explicitly excluded.



### Input fields

#### Required

| Field | Type | Description |
|---|---|---|
| `hour` | int | Hour of booking (0–23) |
| `month` | int | Month of booking (1–12) |
| `is_weekend` | int | 1 if Saturday or Sunday, else 0 |
| `vtat_missing` | int | 1 if no driver was found; `Avg VTAT` must be null in that case |
| `Avg VTAT` | float\|null | Vendor time to assign in minutes (null when `vtat_missing=1`) |
| `day_of_week` | str | Full day name, e.g. `Monday` |
| `Vehicle Type` | str | One of: `Auto`, `Bike`, `eBike`, `Go Mini`, `Go Sedan`, `Premier Sedan`, `Uber XL` |
| `pickup_grouped` | str | Pickup location name or `Other` |
| `drop_grouped` | str | Drop location name or `Other` |

#### Optional — historical features

If omitted or null, each defaults to the dataset's overall cancellation rate (0.38) or 0 for ride counts. Pass these when your system has past data for the customer or location.

| Field | Type | Default | Description |
|---|---|---|---|
| `customer_cancel_rate` | float | 0.38 | This customer's prior cancellation rate (0.0–1.0) |
| `pickup_risk` | float | 0.38 | Historical cancel rate at this pickup location (0.0–1.0) |
| `drop_risk` | float | 0.38 | Historical cancel rate at this drop location (0.0–1.0) |
| `route_risk` | float | 0.38 | Historical cancel rate on this exact route (0.0–1.0) |
| `customer_prior_rides` | int | 0 | Number of rides this customer has completed before |

#### Derived server-side — do not send

`is_peak`, `hour_bucket`, and `vtat_bucket` are computed from `hour` and `Avg VTAT` automatically. Sending them in the request body has no effect.
